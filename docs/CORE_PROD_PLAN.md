# SmartSell — План доведения **ядра** до Production‑Ready (2026‑01‑27)

> Цель документа: зафиксировать, **что считается “ядром”**, что уже готово, какие **дырки мешают продакшену**, и дать пошаговый план работ “одна задача → один результат → проверка → запись в журнал”.

---

## 1) Что мы называем “ядром” SmartSell

**Ядро = то, без чего нельзя безопасно подключать первых клиентов и поддерживать их ежедневно:**
1. **Multi‑tenant безопасность (изоляция компании)** во всех v1 API и storages.
2. **Auth / Users / RBAC**: вход, refresh‑сессии, приглашения, смена/восстановление доступа, базовый аудит.
3. **Биллинг‑контур ядра**: wallet/accounts/ledger, payments intents (хотя бы NoOp/заглушки), subscriptions (базовые статусы/лимиты).
4. **Базовые домены платформы**: Products (CRUD), Campaigns (CRUD/базовый запуск/планировщик через PROCESS_ROLE), Invoices (минимум CRUD).
5. **Операционный минимум продакшена**: миграции, конфиги, healthchecks, логи/ошибки, безопасные дефолты, повторяемый деплой.

**Не‑ядро (интеграции/надстройки)** — подключаем после ядра:
- Kaspi Product Sync/MC cookies/BFF, Feed upload, repricing, advanced analytics, notifications, auto‑dumping, preorder, admin panel и т.п.

---

## 2) Что уже есть (база, на которую опираемся)

Из PROJECT_JOURNAL.md видно, что сделана существенная часть платформенных гарантий:

- **Строгий выбор DB URL (runtime vs test)**, чтобы приложение в dev не “переезжало” на test БД при наличии TEST_* env.
- **Tenant scoping** убран из входных параметров: v1 API больше не доверяет company_id из query/body и использует scoped‑контекст.
- **v1 слой приведён к async‑native** (AsyncSession + await db.get/execute), добавлены guard‑тесты против sync‑DB вызовов.
- **Payments/OTP/Messaging интеграции через resolver + provider configs**, есть Integration Center (конфиги, redaction, healthcheck, eventing).
- **Kaspi orders sync** уже существует как интеграция (инкрементальность, идемпотентность, advisory lock, state endpoint).

---

## 3) Главная причина ощущения “делаем одно и то же”

Мы реально много времени тратили на **операционные блокеры**, а не на новые фичи ядра:

- “Постоянно логинимся” — это симптом, что **smoke‑скрипты и окружение не были стабилизированы**: токены истекали, сервер мог быть не перезапущен с нужными env, маршруты проверялись не через OpenAPI, а в отдельных местах зависимость на Redis/denylist ломала flow.
- Когда нет жёсткого “Definition of Done” на каждый шаг (фикс → тест → запись), ощущается “круг”.

**Вывод:** дальше работаем только по плану ниже — каждый пункт заканчивается измеримым результатом и проверкой.

## 4) План доведения ядра до Production‑Ready

Ниже — список работ в правильном порядке. Каждый пункт имеет: **результат**, **проверку**, **что фиксируем в журнале**.

### Этап A — “Контракт API и стабилизация smoke” (1–3 дня)

**A1. Источник истины по маршрутам = OpenAPI**
- Результат: 1 PowerShell‑скрипт, который:
  - тянет `/openapi.json`,
  - печатает ключевые пути (auth/users/wallet/payments/subscriptions/products/campaigns/invoices/health),
  - падает, если нужного пути нет (чтобы не гонять “curl наугад”).
- Проверка: скрипт проходит на локальном dev.
- Журнал: добавляем “OpenAPI smoke script v1”.

**A2. Единый login/refresh wrapper**
- Результат: 1 файл `scripts/smoke/Login-SmartSell.ps1`, который:
  - логинится один раз,
  - автоматически обновляет access через refresh,
  - кэширует токены в памяти текущей сессии (не в файле).
- Проверка: 10 последовательных вызовов `/api/v1/users/me` без ручного логина.
- Журнал: “smoke auth wrapper стабилен”.

**A3. Стабильный health endpoint**
- Результат: единый endpoint **(и документированный)**:
  - `/api/v1/health` (API‑уровень) и/или `/health` (process‑уровень),
  - без внешних зависимостей “по умолчанию”.
- Проверка: `curl .../api/v1/health` → 200.
- Журнал: “health contract закреплён”.

---

### Этап B — “Схема БД и миграции без дрейфа” (2–4 дня)

**B1. Устранить дрейф `users.hashed_password`**
- Результат: миграция, которая приводит `public.users.hashed_password` к безопасному типу (`TEXT`) и согласует ограничения.
- Проверка:
  - `alembic upgrade head` на чистой БД,
  - ручной `UPDATE users SET hashed_password=...` проходит,
  - логин работает.
- Журнал: “hashed_password TEXT + verified”.

**B2. Политика миграций**
- Результат: правило “старые миграции не редактируем”, любые изменения — новой миграцией. Добавить короткий гайд в docs.
- Проверка: в репо нет правок исторических миграций.
- Журнал: “migration policy locked”.

**B3. Smoke‑тест миграций как обязательный шаг локально**
- Результат: чеклист/скрипт делает:
  - `alembic upgrade head` на smartsell_test,
  - `alembic downgrade -1` (если допускается),
  - снова `upgrade head`.
- Проверка: стабильный прогон.
- Журнал: “migration smoke локально обязателен”.

---

### Этап C — “Auth/RBAC как продакшен‑система” (3–6 дней)

**C1. 401/403 никогда не превращаются в 500**
- Результат: `/api/v1/auth/me` и `/api/v1/users/me` при отсутствии/битом/истёкшем токене всегда возвращают 401/403, а не 500.
- Проверка: автотесты + smoke.
- Журнал: “auth error propagation fixed”.

**C2. Logout‑семантика**
- Результат: определить и закрепить правило:
  - logout “гасит” только refresh‑сессию, или
  - logout также “ревокает” access (denylist).  
  Поведение документировать и покрыть тестом.
- Проверка: smoke сценарии.
- Журнал: “logout semantics locked”.

**C3. Админ‑bootstrap без OTP**
- Результат: супер‑админ создаёт первых пользователей и подключает магазины без OTP; остальным OTP включаем только при готовом провайдере.
- Проверка: тесты/ручной сценарий.
- Журнал: “admin bootstrap no‑otp verified”.

---

### Этап D — “Биллинг‑контур ядра” (4–8 дней)

**D1. Wallet как бухгалтерский источник**
- Результат:
  - ledger неизменяемый (append‑only),
  - баланс считается детерминированно,
  - операции идемпотентны (client_request_id).
- Проверка: тесты + небольшой нагрузочный сценарий.
- Журнал: “wallet ledger invariants”.

**D2. Payments intents (минимум)**
- Результат:
  - единый интерфейс payment gateway,
  - NoOp остаётся, но контракт готов для TipTop/др.
- Проверка: тесты provider resolver.
- Журнал: “payments contract stable”.

**D3. Subscriptions**
- Результат:
  - состояния подписки, период, заморозка/отмена,
  - enforcement (feature flags/лимиты) — хотя бы каркас.
- Проверка: tenant tests.
- Журнал: “subscriptions enforcement skeleton”.

---

### Этап E — “Базовые домены платформы” (4–10 дней)

**E1. Products**
- Результат: CRUD + валидации (SKU/slug/цены/остаток) + tenant scoping.
- Проверка: тесты + smoke.
- Журнал: “products ready”.

**E2. Campaigns**
- Результат: CRUD + минимальный запуск через PROCESS_ROLE (без дублей).
- Проверка: тесты + локальный запуск.
- Журнал: “campaigns runner stable”.

**E3. Invoices**
- Результат: довести до минимально полезного:
  - статусы,
  - связка с платежами/кошельком,
  - подготовка к PDF как интеграции позже.
- Проверка: тесты.
- Журнал: “invoices MVP core”.

---

### Этап F — “Production packaging” (3–7 дней)

**F1. Конфиги**
- Результат: `.env.example` актуальный; все опции безопасны по умолчанию.
- Проверка: запуск с минимальным набором env.
- Журнал: “env contract”.

**F2. Логи/ошибки**
- Результат:
  - request_id в логах,
  - единый формат ошибок (code/detail),
  - никакой утечки секретов.
- Проверка: тесты + grep по логам.
- Журнал: “logging hardened”.

**F3. Deploy‑гайд**
- Результат: 1 документ “как поднять” (1 VM):
  - Postgres,
  - Uvicorn/Gunicorn,
  - reverse proxy,
  - миграции,
  - бэкапы (минимум регламент).
- Проверка: dry‑run по инструкции.
- Журнал: “deploy playbook”.

## 5) Как будем работать по плану (чтобы не было ощущения “круга”)

Правило на каждый шаг:
1) Берём **один пункт** из плана.  
2) Фиксируем ожидаемый результат (что должно измениться).  
3) Делаем изменения.  
4) Запускаем проверку (pytest/ruff/smoke).  
5) Записываем 5–10 строк в PROJECT_JOURNAL.md: “что сделали / как проверили / что дальше”.

---

## 6) “Definition of Done” для ядра

Ядро считается **готовым к prod‑pilot**, если:
- миграции применяются на чистую БД и на существующую без ручных правок;
- auth/users стабильны (refresh, приглашения, восстановление, корректные 401/403);
- tenant isolation доказан тестами на ключевых доменах;
- wallet/payments/subscriptions работают и не ломают бухгалтерскую целостность;
- есть повторяемый smoke‑набор и deploy‑гайд.
